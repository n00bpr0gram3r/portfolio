3:T56b,<h1>SSRF on Media Ingestion Pipeline via Redirect</h1>
<h2>Vulnerability Summary</h2>
<p>During a security audit, we identified that the platform&#39;s video ingestion engine followed HTTP 302 redirects to internal networks. The downloader did not enforce target IP filters, allowing connections to local loopback ports (<code>127.0.0.1</code>), Kubernetes Service Account tokens, and cloud metadata endpoints (<code>169.254.169.254</code>).</p>
<h2>Technical Breakdown</h2>
<p>The target endpoint accepting video URLs for transcoding allows users to import remote video files.</p>
<h3>1. The Redirection Trick</h3>
<p>By providing a URL to an external server we control, we can return a HTTP 302 redirect payload:</p>
<pre><code class="language-http">HTTP/1.1 302 Found
Location: http://169.254.169.254/latest/meta-data/
</code></pre>
<h3>2. Network Boundary Bypass</h3>
<p>The server-side worker resolved the redirect internally, bypassing corporate network firewalls and requesting the sensitive AWS IMDS metadata endpoints.</p>
<h2>Remediation &amp; Hardening</h2>
<ol>
<li><strong>Disable Redirects by Default:</strong> Disable redirect-following inside the HTTP downloader library configuration.</li>
<li><strong>Egress Subnet Filtering:</strong> Move transcode workers to an isolated private subnet with strict egress rules blocking all RFC 1918 and link-local ranges.</li>
</ol>
